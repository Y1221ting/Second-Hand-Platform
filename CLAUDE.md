## 中断处理：中途改需求怎么办

开发过程中，如果我突然说「先停一下，我要改个别的」，按以下流程走：

### 1. 先停下当前工作
- 如果当前已有一半的改动还没提交，先问我要不要提交当前进度（防止丢代码）
- 我说「先放着」就不提交，我说「先提了」就正常 commit

### 2. 处理临时需求
- 按同样的「方案→确认→执行→推送→部署测试」流程处理这个临时功能
- 不允许因为"是临时加的"就跳过步骤

### 3. 处理完后提醒我回归正轨
明确说一句：
```
临时功能「XXX」已完成。现在可以回到原来的阶段了，下一步是：<原计划的下个功能点>
```
- 必须帮我回忆起当前做到哪一步了
- 如果我忘了原来的计划，你要根据之前的对话帮我回忆

## 工作流程（严格五步走）

### 第一步：先出方案，给我看
涉及以下操作，必须先输出方案，等我确认后才能动手：
- 任何代码改动（新增、修改、删除）
- 修改数据库结构
- 修改配置/部署
- 重构代码

**方案输出格式**：
```
## 改造方案：<功能名>
### 做什么
（一句话说清楚要改什么、为什么改）

### 涉及文件
- xxx.js — 修改什么
- xxx.js — 修改什么

### 改动逻辑
（解释为什么要这么改，原理是什么）

### 执行步骤
1. ...
2. ...

### 风险点
- ...
```

### 第二步：我确认后，逐个小步骤执行
- 每改完一个小功能点就停下来，告诉我改完了
- 不要一次性改完三四个功能才说

### 第三步：先本地构建，再提交推送
**顺序：先 `npm run build`，构建成功后再 commit + push**。如果构建失败，先修复再提交，绝不把构建失败的代码推上去。

格式一律为：
```
cd Client && npm run build       # 先本地构建，确认无错误
git add -A
git commit -m "功能名：具体改了啥"
git push
```

**重要规则**：
- ✅ 改完一个小功能就提交一次，不积攒
- ✅ 每次推送前先 `npm run build`，构建成功才能推
- ✅ commit message 写清楚改了啥，方便回滚定位
- ❌ 不要攒一堆才推——出问题回溯会把好的功能也搞丢
- ❌ 绝不跳过本地构建——构建失败可能是漏文件、语法错误，服务器上再发现就晚了

### 第四步：通知可部署 + 给出测试方法
代码推送后，你需要告诉我两件事：

**1. 可以部署了**
- 明确说「代码已推送，可以在服务器部署了」
- 列出部署命令（我直接复制粘贴）

**2. 部署后怎么测试**
- 告诉我打开哪个页面、做什么操作来验证功能生效
- 例如：「打开 http://freevian.top/home，搜索 XXX，应该能看到...」
- 如果有后端接口要测试，也给出 curl 命令

### 第五步：记录改动
每次改动后，在当前会话末尾记录：
```
## 改动记录
### [时间] 改动：<功能名>
- **涉及文件**：xxx.js
- **改了什么**：...
- **为什么改**：...
```

## 沟通风格
- 像个带徒弟的工程师，解释清楚每一步的意图和原理
- 可以给选项说明优劣，但最终由我决定
- 不做任何没经过我确认的改动
