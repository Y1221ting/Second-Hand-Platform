# 求购模块优化方案

> 校园 C2C 二手交易平台 · 同学求购功能独立路由 + 体验升级
> 日期：2026-06-08 | 分支：refactor/v3-simplify

---

## 一、现状与问题诊断

### 1.1 当前架构

```
AddProduct.js（发布页）
├── tab: 我要卖 ── 完整出售表单（~300 行）
├── tab: 我要买 ── 3 字段求购表单（~60 行 夹在中间）
└── 切换逻辑加重复的条件渲染

Home/index.js（首页）
├── WantedList ── 硬编码 limit=4 的求购卡片区块
└── 多一次网络请求 /api/wanted?limit=4

WantedList.js（28 行）
├── 仅展示：名称 + 预算 + 描述(2行截断) + 发布人
└── 无任何交互：不能联系、不能展开、不能操作
```

### 1.2 核心问题

| 维度 | 问题 | 严重程度 |
|------|------|----------|
| **发现** | 求购入口藏在「发布」页的第二个 tab，用户不知道 | 🔴 致命 |
| **行动** | 卡片只展示信息，没有任何联系/回应机制 | 🔴 致命 |
| **转化** | 卖家看不到求购（首页只展 4 条，且不参与搜索） | 🟡 严重 |
| **管理** | 发完求购后无法删除/标记已买到 | 🟡 严重 |
| **信任** | 不知道求购是什么时候发的，旧帖无法识别 | 🟢 一般 |
| **维护** | 求购表单混在 AddProduct.js 中，增加维护成本和构建风险 | 🟢 一般 |

## 二、总体方案：独立路由 + 简化归位

### 2.1 核心决策

```
❌ 旧方案：发布页 tab 切换「我要卖」/「我要买」
✅ 新方案：求购独立路由 /wanted + AddProduct 只保留出售
```

### 2.2 路由设计

```
/wanted          → 求购广场（浏览全部 + 发布入口 + 我的求购）
/add-product     → 仅出售表单（删掉 buy tab）

首页 Home         → 去掉 WantedList 区块，替换为一行的轻量入口 banner
```

## 四、每个优化的详细方案

### 优化 1：联系方式字段（P0 — 功能生死线）

**问题**：求购发出去后，看到的人无法联系发布者，功能等于无效。

**方案**：
- Model 加 `contact: { type: String, default: "" }`（选填，用户填微信/QQ/手机号）
- 后端 POST 接口透传该字段
- 求购表单加显式输入框：「联系方式（选填）— 微信/QQ/手机号」
- 卡片展示联系人信息，点击「联系TA」弹窗或直接展示
- 公共列表 API 不再删除 phone（因为 contact 是用户主动填的，不存在隐私泄露）

**预期效果**：从「只能看」变成「能联系」，功能走通闭环

**改动量**：3 文件（Model + Route + Page）

---

### 优化 2：我的求购 — 删除功能（P0 — 体验闭环）

**问题**：用户发完求购后无法撤销，买到了还挂在首页。

**方案**：
- 后端加 `DELETE /api/wanted/:id`（验证 `postedBy.id` 匹配当前用户）
- 前端在「我的求购」区域每张卡片右下角加「已买到」按钮
- 删除成功从列表移除，不重新加载全部

**不动 status 字段**：删除即从数据库移除，不保留僵尸数据。

**预期效果**：用户对发布内容有掌控感

**改动量**：2 文件（Route + Page）

---

### 优化 3：首页替代 WantedList（减负）

**问题**：Home 页加载 WantedList 组件，多一次网络请求，布局复杂，且只露出 4 条意义不大。

**方案**：
- 删除 `Home/index.js` 中的 `<WantedList />` 和引用
- 在原位置替换为一行的轻量文字链接：

```
📢 同学在求购 N 件商品  [去看看 →]
```

- 点击后跳转到 `/wanted`
- N 来自一个轻量请求 `GET /api/wanted/count` 或直接在 `GET /api/wanted?limit=1` 的响应头中带 total

**预期效果**：
- Home 页减少一次数据渲染，首屏更快
- 求购有独立阵地，不再被商品列表挤压
- 用户被引导进入 `/wanted`，而不是在 Home 页浅尝辄止

**改动量**：2 文件（Home/index.js 修改 + WantedList.js 删除）

---

### 优化 4：相对时间显示（P1 — 信任感）

**方案**：纯前端，在 WantedPage 卡片渲染时对 `createdAt` 做相对时间转换。

```
< 1 分钟 → "刚刚"
< 60 分钟 → "N 分钟前"
< 24 小时 → "N 小时前"
< 7 天 → "N 天前"
≥ 7 天 → "N 月 N 日"
```

**不引入第三方库**，手写一个 15 行的 `timeAgo` 工具函数。

**预期效果**：浏览者一眼判断需求时效性，提升联系意愿

**改动量**：1 文件（WantedPage.js 内联函数）

---

### 优化 5：发布即匹配提示（P1 — 核心价值场景）

**问题**：用户发布求购后不知道有没有人卖，不知道这个功能到底有没有用。

**方案**：发布成功后，查询同分类商品数量，展示：

```
✅ 求购已发布！

当前有 12 件「教材教辅」类商品在售
去看看有没有你想要的？[浏览商品 →]
```

**技术实现**：
- 后端 POST 接口发布成功后，加一个可选查询：根据求购内容（未来加分类字段后），返回同类商品数量
- 或者简单版本：前端收到成功响应后，调 `GET /api/products/counts?category=xxx`
- 最简单的 MVP：发布成功后，固定显示「去首页看看有没有你想要的 [去逛逛]」

**预期效果**：
- 用户发布后看到「有人有」，信心大增
- 引导用户从求购切换到浏览商品，形成「发完就逛」的习惯

**改动量**：1 文件（WantedPage.js 发布成功回调逻辑）

---

### 优化 6：发布入口 + 模板化快速发布（P1 — 降低发布阻力）

**问题**：求购表单是空白填空式，用户不知道填什么、怎么填。

**方案**：
- 表单上方加分类模板按钮：[📚 教材] [📱 数码] [👕 服饰] —— 点击后自动填入描述模板
- 描述输入框下方加快速描述按钮：[九成新就行] [急用] [想收个二手]
- 联系方式字段加 placeholder 引导：「填了别人才好联系你哦」

**预期效果**：从填表变成选模板，发布率提升

**改动量**：纯前端，WantedPage.js 内完成

---

### 优化 7：冷启动空状态设计（P1 — 挽留新用户）

**方案**（三态）：

| 场景 | 展示 |
|------|------|
| 无任何求购 | ❄️ 大空态：「还没有同学发布求购」+ 大大的「发布第一条」按钮 |
| 筛选后无结果 | 🧐 「换个关键词试试？」+ 清除筛选按钮 |
| 我的求购为空 | 💡 「发布求购后，会在这里显示」+ 引导发布 |

**改动量**：纯 JSX 条件渲染

---

### 优化 8：查看更多 — 分页加载（P2 — 曝光放大）

**方案**：
- 后端 GET 加 `page` / `skip` / `limit` 完整分页参数（当前只支持 `limit`）
- 前端列表底部，「加载更多」按钮，点击追加下一页
- 数据累加，不替换

**预期效果**：从 4 条限制变为无限滚动，卖家可以完整浏览所有需求

**改动量**：2 文件（Route + Page）

---

### 优化 9：热门求购 Tag（P2 — 引导供需）

**方案**：
- 在全部求购列表上方，显示预设的热门标签
- 冷启动阶段用静态数据，后期数据量上来后改为真实统计
- 点击 Tag 在列表顶部做关键词高亮/筛选

```
🔥 热门求购
[高数课本] [二手手机] [台灯] [电动车] [四六级资料]
```

**预期效果**：
- 引导卖家：「这么多人要 X，我可以卖」
- 营造社区活跃感
- 冷启动时「作弊」让页面看起来有内容

**改动量**：纯前端静态数据，WantedPage.js 内完成

---

### 优化 10：一键联系 + 预设话术（P2 — 提升转化）

**方案**：点击「联系TA」后弹窗展示：

```
联系方式
─────────────────
📱 微信：wang_123
📞 电话：138xxxxxxx

「同学你好，我在校园二手市场看到你求购
「高数课本」，我正好有，方便联系吗？」

[复制话术] [已复制，去微信联系]
```

**设计意图**：
- 降低用户「不知道怎么开口」的心理门槛
- 复制即用，无缝跳转到微信完成沟通
-无需平台做即时通讯

**改动量**：纯前端，WantedPage.js 内弹窗组件

## 五、不要做的（克制清单）

| 功能 | 原因 |
|------|------|
| 站内即时聊天 | 2核2G 服务器扛不住 WebSocket；校园场景微信已覆盖沟通 |
| 求购图片上传 | 增加存储 + 审核 + CDN 负担，小功能不值得 |
| 匿名发布 | 校园实名更有信任感，匿名增加沟通成本 |
| 议价/竞价 | C2C 二手不需要，太复杂 |
| 自动通知匹配 | 需要轮询/WebSocket，当前资源不支持 |
| 审核求购内容 | 滥用概率低，出现后再加 |
| 求购独立分类模型 | 先用字符串，量大了再考虑独立分类 |

## 六、后端 API 变更一览

### 新增接口

```
DELETE /api/wanted/:id          — 删除自己的求购（authMiddleware 保护）
GET    /api/wanted/mine         — 获取自己的求购列表（authMiddleware 保护）
```

### 修改接口

```
GET /api/wanted
  - 新增 query 参数：page (默认1), limit (默认20)
  - 返回格式改为 { wanteds: [], total: number, page: number, totalPages: number }
```

### 模型变更

```js
// Wanted.js 新增字段
contact: { type: String, default: "" },   // 联系方式（选填）
status: { type: String, enum: ["active", "fulfilled"], default: "active" }
// 注：status 先加上但不强依赖，删除操作直接删 document
```

## 八、风险与注意事项

| 风险 | 应对 |
