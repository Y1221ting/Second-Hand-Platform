## 模块二：HTTPS 配置（P1）

> ⏱ 预计耗时：1-2 小时

### 2.1 方案选择：acme.sh + Let's Encrypt

**为什么不用 certbot**：certbot 依赖 Python 环境且较重，acme.sh 是纯 shell 脚本，资源占用极低，更适合 2 核 2G 小机器。

### 2.2 安装与签发

```bash
# SSH 到服务器，root 用户执行

# 1. 安装 acme.sh
curl https://get.acme.sh | sh -s email=你的邮箱@qq.com
source ~/.bashrc

# 2. 停止 Nginx（acme.sh 需要 80 端口做验证）
systemctl stop nginx

# 3. 签发证书（standalone 模式，不需要 Nginx）
~/.acme.sh/acme.sh --issue -d freevian.top --standalone

# 4. 创建证书存放目录
mkdir -p /etc/nginx/ssl

# 5. 安装证书
~/.acme.sh/acme.sh --install-cert -d freevian.top \
  --key-file       /etc/nginx/ssl/freevian.top.key \
  --fullchain-file /etc/nginx/ssl/freevian.top.crt \
  --reloadcmd      "systemctl reload nginx"

# 6. 证书每 60 天自动续期（acme.sh 会添加 cron）
```

### 2.3 Nginx HTTPS 配置

编辑 `/etc/nginx/sites-available/freevian.top`：

```nginx
# HTTP → HTTPS 强制跳转
server {
    listen 80;
    server_name freevian.top;
    return 301 https://$host$request_uri;
}

server {
    listen 443 ssl http2;
    server_name freevian.top;

    # 证书
    ssl_certificate     /etc/nginx/ssl/freevian.top.crt;
    ssl_certificate_key /etc/nginx/ssl/freevian.top.key;

    # 安全协议（禁用老旧 TLS）
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384;
    ssl_prefer_server_ciphers off;
    ssl_session_cache shared:SSL:10m;
    ssl_session_timeout 10m;

    # 安全头
    add_header Strict-Transport-Security "max-age=63072000; includeSubDomains; preload" always;

    client_max_body_size 50m;

    # /api/ 和 /uploads/ → 后端容器
    location /api/ {
        proxy_pass http://127.0.0.1:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    location /uploads/ {
        proxy_pass http://127.0.0.1:8000;
        proxy_set_header Host $host;
    }

    # 其余请求 → 前端容器
    location / {
        proxy_pass http://127.0.0.1:5000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

```bash
# 启用配置
ln -sf /etc/nginx/sites-available/freevian.top /etc/nginx/sites-enabled/
nginx -t && systemctl start nginx

# 验证
curl -I https://freevian.top
# 应该看到 HTTP/2 200 和 Strict-Transport-Security 头
```

## 模块三：生产环境加固（P1）

> ⏱ 预计耗时：2-3 小时

### 3.1 服务器防火墙

```bash
# SSH 到服务器，root 执行

# 1. 安装并启用 ufw
apt-get install -y ufw

# 2. 默认策略：拒绝所有入站，允许所有出站
ufw default deny incoming
ufw default allow outgoing

# 3. 只开放必要的端口
ufw allow 22/tcp        # SSH
ufw allow 80/tcp        # HTTP
ufw allow 443/tcp       # HTTPS

# 4. 明确拒绝 27017（MongoDB）和其他敏感端口
ufw deny 27017
ufw deny 8000           # 后端端口不需要对外
ufw deny 5000           # 前端端口（走 Nginx 80/443）

# 5. 启用
ufw enable
ufw status verbose

# 6. Docker 会绕过 ufw，加 iptables 兜底
iptables -I DOCKER-USER -p tcp --dport 27017 -j DROP
iptables -I DOCKER-USER -s 127.0.0.1 -p tcp --dport 27017 -j ACCEPT
iptables -I DOCKER-USER -p tcp --dport 8000 -j DROP
iptables -I DOCKER-USER -p tcp --dport 5000 -j DROP

# 7. 持久化
apt-get install -y iptables-persistent
netfilter-persistent save
```

### 3.2 SSH 加固

```bash
# 编辑 /etc/ssh/sshd_config，修改以下配置：

# 禁用 root 密码登录（改用密钥）
PermitRootLogin prohibit-password

# 禁用密码认证（全部用密钥）
PasswordAuthentication no

# 修改 SSH 端口（降低扫描风险）
Port 2222    # 换一个不常用的端口，改了后记得 ufw allow 2222

# 重启 SSH
systemctl restart sshd
```

> 如果密钥配置麻烦，至少保留密码登录但限制 root 密码强度。

## 模块四：监控与告警（P2）

### 4.1 轻量级方案：Node Exporter + 自写健康检查脚本

**为什么不用 Prometheus + Grafana**：那是吃内存的大户（Grafana 单进程 ~200MB），不适合 2G 小机器。用最简单的方案：

```bash
# 方案：cron 定时脚本 + 邮件告警（用 QQ 邮箱 SMTP）

# 1. 创建监控脚本 /opt/monitor.sh
cat > /opt/monitor.sh << 'SCRIPT'
#!/bin/bash
# 轻量健康监控 — 每 5 分钟执行一次

WEB_URL="https://freevian.top"
API_URL="https://freevian.top/api/health"
ALERT_EMAIL="你的邮箱@qq.com"
TMPFILE="/tmp/monitor_$(date +%s).txt"

# 检查网站可访问性
HTTP_CODE=$(curl -s -o /dev/null -w "%{http_code}" -m 10 "$WEB_URL")
if [ "$HTTP_CODE" != "200" ]; then
  echo "网站不可访问，HTTP状态码: $HTTP_CODE" >> "$TMPFILE"
fi

# 检查健康检查接口
API_CODE=$(curl -s -o /dev/null -w "%{http_code}" -m 10 "$API_URL")
if [ "$API_CODE" != "200" ]; then
  echo "后端健康检查失败，HTTP状态码: $API_CODE" >> "$TMPFILE"
fi

# 检查内存使用率
MEM_USAGE=$(free | grep Mem | awk '{printf "%.0f", $3/$2 * 100}')
if [ "$MEM_USAGE" -gt 85 ]; then
  echo "内存使用率过高: ${MEM_USAGE}%" >> "$TMPFILE"
fi

# 检查磁盘使用率
DISK_USAGE=$(df / | tail -1 | awk '{print $5}' | tr -d '%')
if [ "$DISK_USAGE" -gt 85 ]; then
  echo "磁盘使用率过高: ${DISK_USAGE}%" >> "$TMPFILE"
fi

# 检查 Docker 容器运行状态
for container in second-hand-mongodb second-hand-backend second-hand-frontend; do
  if ! docker ps --format '{{.Names}}' | grep -q "$container"; then
    echo "Docker 容器未运行: $container" >> "$TMPFILE"
  fi
done

# 发邮件告警（如果部署了邮件服务）
if [ -s "$TMPFILE" ]; then
  # 写入错误日志
  cat "$TMPFILE" >> /var/log/monitor.log
fi
rm -f "$TMPFILE"
SCRIPT

chmod +x /opt/monitor.sh

# 2. 添加 cron，每 5 分钟执行
echo "*/5 * * * * /bin/bash /opt/monitor.sh" | crontab -
```

### 5.2 备份异地存储（重要！）

备份只存服务器本地 = 服务器硬盘坏了全丢。需要异地存储：

**方案一：scp 到备用服务器（免费，推荐）**
```bash
# 备用服务器 47.96.17.173 上有 2 核 2G，可以存备份

# 1. 配置 SSH 密钥免密登录

# 2. 修改 backup.sh，末尾追加远程同步：
rsync -avz --remove-source-files \
  -e "ssh -i ~/.ssh/backup_key" \
```

## 模块六：CI/CD 流水线（P2）

> ⏱ 预计耗时：2-3 小时 | 🔄 上线后 1-2 周内

### 6.1 GitHub Actions 自动构建 + 手动部署

考虑到 2 核 2G 服务器不适合在服务器上 build（`npm run build` 会吃满 CPU 让服务不可用），使用 GitHub Actions 在云端构建，然后 SSH 到服务器拉取。

```yaml
# .github/workflows/deploy.yml
name: Build and Deploy

on:
  push:
    branches: [main]  # 上线后切换到 main 分支

jobs:
  build-frontend:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 18
          cache: 'npm'
          cache-dependency-path: Client/package-lock.json

      - name: Install dependencies
        working-directory: Client
        run: npm ci

      - name: Build
        working-directory: Client
        run: npm run build
        env:
          CI: false  # CRA 默认 CI=true 会把 warning 当 error

      - name: Upload build artifact
        uses: actions/upload-artifact@v4
        with:
          name: frontend-build
          path: Client/build/

  # 说明：真正的自动部署需要把 build 产物传到服务器。
  # 简单方案：GitHub Actions build → commit build 到仓库 → 服务器 git pull → docker compose up -d
  # 复杂方案：rsync/scp 到服务器（需要配置 SSH secrets）
```

### 6.2 当前实用方案：本地 build + git push + 服务器 pull

这是目前最贴合实际且零成本的方案：

```bash
# === 开发机（Windows） ===
cd Client
npm run build        # 本地构建（快，不影响服务器）
# 确认 build/ 目录已生成

# 提交构建产物
git add Client/build/
git commit -m "build: 前端构建产物 $(date +%Y%m%d)"
git push

# === 服务器 ===
ssh root@freevian.top
cd /www/wwwroot/Second-Hand-main
git pull
docker compose up -d --no-deps frontend   # 只重建前端
```

### 6.3 回滚策略

```bash
# 快速回滚到上一个版本
git log --oneline -5          # 找到要回滚的 commit
git revert <commit-hash>      # 安全回滚（保留历史）
git push
# 服务器 git pull + docker compose up -d

# 紧急回滚（直接切到上一个版本）
git reset --hard HEAD~1
git push --force              # ⚠️ 谨慎使用
```

## 模块九：应急预案（P0）

> ⏱ 预计耗时：1 小时（熟悉流程 + 准备联系方式）| 🔴 上线前必须熟悉

### 9.1 故障场景矩阵

| 故障 | 概率 | 影响 | 恢复时间 |
|------|------|------|----------|
| 服务器宕机 | 低 | 全站不可用 | 10-30 分钟 |
| MongoDB 挂了 | 低 | 数据不可读写 | 5-15 分钟 |
| 内存/磁盘满了 | 中 | 服务变慢或不可用 | 5-20 分钟 |
| 域名 DNS 问题 | 极低 | 域名不可达 | 取决于 DNS 服务商 |
| 被恶意攻击 | 低 | 服务不可用 | 30 分钟 - 数小时 |
| 代码 Bug 导致 500 | 中 | 部分功能不可用 | 5-30 分钟 |

### 9.2 场景一：服务器宕机

**症状**：`curl http://freevian.top` 无响应，SSH 连不上。

**处理步骤**：
```bash
# 1. 登录阿里云控制台 → ECS 实例 → 查看实例状态
# 2. 如果实例停止 → 启动实例
# 3. 如果能 ping 通但 SSH 连不上 → 用阿里云 VNC 登录
# 4. 登录后：
systemctl restart docker
cd /www/wwwroot/Second-Hand-main
docker compose up -d

# 5. 验证
curl http://localhost:8000/api/health
curl http://localhost:5000/
```

### 9.3 场景二：MongoDB 挂了

**症状**：所有需要数据的 API 返回 500。

**处理步骤**：
```bash
# 1. 查看 MongoDB 容器状态
docker ps | grep mongo

# 2. 如果容器停止了：
docker start second-hand-mongodb

# 3. 查看日志定位原因：
docker logs --tail 50 second-hand-mongodb

# 4. 如果数据损坏，走恢复流程：
# 参照模块五 → 5.3 恢复演练

# 5. 如果 MongoDB 反复崩溃，可能是内存不足：
# 降低 cacheSizeGB（docker-compose.yml）
docker compose down
# 编辑 docker-compose.yml：--wiredTigerCacheSizeGB 0.15
docker compose up -d
```

### 9.4 场景三：磁盘/内存满了

**症状**：网站变慢、上传图片失败、后端报错。

**处理步骤**：
```bash
# 诊断
df -h                          # 看磁盘
free -h                        # 看内存
docker stats --no-stream       # 看各容器使用情况

# 清理磁盘（由易到难）
docker system prune -f         # 清理 Docker 垃圾
rm -rf /var/log/*.gz           # 清理旧系统日志
# 检查 uploads 目录是否有超大文件
find /www/wwwroot/Second-Hand-main/Server/uploads -size +10M

# 清理内存
docker compose restart backend  # 重启后端释放 Node 内存
```

### 9.5 场景四：被恶意攻击

**症状**：CPU 100%、API 返回大量 429、陌生 IP 大量请求。

**处理步骤**：
```bash
# 1. 查看当前请求量
tail -200 /var/log/nginx/access.log | awk '{print $1}' | sort | uniq -c | sort -rn | head -10

# 2. 封锁攻击 IP（找到 TOP 恶意 IP 后）
iptables -A INPUT -s <恶意IP> -j DROP

# 3. 如果扛不住，临时启动更高强度限流
# 在 server.js 中临时把 100/min 改为 30/min 并重新部署

# 4. 如果是 DDoS，联系阿里云开启高防（需要额外购买）

# 5. 事后分析：检查攻击来源和目标接口
```

### 9.6 场景五：用户数据泄露

**症状**：发现批量用户信息被爬取、或数据库被拖库。

**处理步骤**：
```bash
# 1. 立即通知用户更改密码
# （通过系统消息或邮件告知所有用户）

# 2. 强制所有用户 token 失效
# 在 MongoDB 中批量递增 tokenVersion：
docker exec second-hand-mongodb mongosh -u admin -p <密码> \
  --authenticationDatabase admin second-hand \
  --eval "db.users.updateMany({}, {\$inc: {tokenVersion: 1}})"

# 3. 检查泄露源头
# - 查看 Nginx access log，寻找异常请求模式
# - 检查是否有接口没加权限校验
# - 检查 .env 是否被提交到 git

# 4. 修复漏洞
# 5. 更换 JWT_SECRET（所有用户需重新登录）
# 6. 更换 MongoDB 密码
```
